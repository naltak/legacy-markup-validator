W3C HTML Validation Service: To Do List

This page has the to-do list for the W3C HTML Validation Service, including bugs that need fixing and general wishlist items.

See also: www-validator mailing list archives for recent discussion that may not be reflected on this page.

These items are roughly in prioritized order; i.e. the items near the top are those which I consider most important.

  1. put error explanations online, re-link them from validator output
  2. apply Takuya's XML encoding patch; merge in Terje's cleanup patch
  3. bug fix: this doc is invalid, but not reported as such
  4. don't output a bazillion errors for documents with bogus FPIs (see a sample)
  5. add a doctype-overriding option (use Terje's patch
  6. add a textarea for testing short HTML fragments
  7. add a "fix my HTML for me" option using tidy.
  8. finish adding support for XML validation (see also: test cases, or an existing service)
  9. update sgml-lib.tar.gz; automate the updates
  10. always display source code if there are errors? (or maybe just show a few lines of context for each error?) Or add a "show source iff errors" option?
  11. allow twiddling of options from the results page
  12. make lists of "most frequently validated invalid pages" and "most frequently validated (non-W3C?) valid pages" (need to start logging stuff first, including IPs to compare uniqueness)
  13. create a stylesheet and link to it from every page
  14. add a section to the report on document cacheability? (or just link to it?)
  15. put the explanations in a database (flat files are probably okay), offer an option to display them inline with the errors
  16. add support for more character encodings
  17. install and play with HTML::Validator, link to it from somewhere: http://www.hut.fi/~samzait/HTML-Validator/ http://www.cpan.org/modules/by-authors/id/S/SA/SAIT/
  18. write documentation, describing each feature and option of the validator and answering questions like "What's the difference between an SGML parser and Weblint?", "Which DOCTYPE should I use?", content negotiation, ...
  19. upgrade weblint to the most recent version
  20. incorporate patches to make it work on win9x systems
  21. incorporate CSS validation directly into this service
  22. make e.g. http://validator.w3.org/check/referer;imgonly return only an image showing the validation status of the referring page
  23. add link validation using Hugo's checklink code (or other code with similar functionality)
  24. give errors/warnings related to markup that is technically valid SGML, but error prone, such as things found in "B.3 SGML implementation notes" in the HTML 4.0 spec. (these things really belong in weblint; does it handle any/all of them?)
  25. make an "elements found" section a la Webtechs, with links from each element to the appropriate place in either the DTD tree listing produced with dtd2html (after running dtd2html with all DTDs in the catalog), or the HTML 3.2/4.0 specs, or htmlhelp.com stuff, ... Similarly, put links on each element in the parse tree.
  26. add a "document meta-information" section to the report, to encourage people to use META tags appropriately?
  27. add a "recommend a DTD for me" feature (check a document against all available DTDs, report which one has the fewest errors)
  28. start caching validation results locally and doing an If-Modified-Since HTTP request to only download and re-validate URLs if they actually changed since their last validation
  29. site walker/validator: need to add a "registered user" feature first, because this feature could be abused (many requests on a server in a short period of time)?
  30. URL-minder service: "remind me if this page or set of pages ever ceases to validate"
    • "registered user" feature is also necessary for this (to prevent unwanted e-mail)
    • "registered users" could have a list of URLs they're interested in, and whenever they return to the service they can modify this list, and e-mail can be sent whenever any of them cease to validate
    • right now someone could probably use the existing URL-minder service instead of writing a new one (tell it to "mind" the URL that points to the validation result for a page?) But I'm not sure how regularly URL-minder checks for changes; it seemed to be weekly or something, which isn't frequent enough, IMO. Doing an If-Modified-Since GET every day doesn't cost much if pages don't change. Some of these features should only be enabled if the page consistently returns a Last-Modified header, maybe.
  31. add an HTML pretty-printer feature, using tidy?
  32. provide messages in different languages?
  33. add a graphical representation of the document's structure, using GIFs-mixed-with-text, or an entire GIF?
  34. add a section with PICS info?
  35. issue PICS labels for documents that do/don't conform? or for editing tools that don't conform?
